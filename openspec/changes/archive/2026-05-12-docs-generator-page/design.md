## Context

GAP already has the full backend pipeline for Mode 3: `parseBackendRoutes` → `classifyFeatures` → `buildDocumentedRoutes` → `generateApiDocs` (in `lib/generators/api-docs.ts`). The `POST /api/docs` route may or may not exist depending on whether proposal #8 wired it up. The frontend has no page for this flow at all. The design system mandates no border-radius, monochrome palette, square spinner, and Space Grotesk / Geist / Geist Mono typography. No external state management; localStorage is acceptable for ephemeral persistence across reloads.

## Goals / Non-Goals

**Goals:**
- Deliver a complete `/docs-generator` page that drives the Mode 3 pipeline end-to-end
- Render Markdown documentation with syntax-highlighted code blocks and per-block copy buttons
- Provide a sticky two-column layout: TOC sidebar (30%) + Markdown viewer (70%)
- Allow export as raw Markdown and OpenAPI JSON download
- Persist last result to `localStorage` so a hard reload does not lose generated docs
- Ensure the page works without a database — if DB is unavailable, the API falls back to returning the result without persisting

**Non-Goals:**
- Authentication or per-user doc history (that is proposal #11)
- GitHub URL input for Mode 3 — paste-only for this proposal
- Real-time streaming of generation progress — polling or a single await is sufficient
- PDF export

## Decisions

### D1 — Markdown renderer: `react-markdown` + `rehype-highlight`

**Decision:** Install `react-markdown` with `rehype-highlight` (uses highlight.js under the hood) for rendering the Markdown string returned by the API.

**Rationale:** `react-markdown` is the de-facto React Markdown renderer. `rehype-highlight` adds syntax highlighting to fenced code blocks with zero additional wiring. The alternative (`marked` + manual `dangerouslySetInnerHTML`) has XSS risk and is harder to extend. `prismjs` is also viable but requires more manual setup.

**highlight.js theme:** Use the `github-dark` theme CSS, imported in `globals.css` under a `.hljs-wrap` scoping class so it doesn't bleed into non-doc sections.

### D2 — TOC generation: derive from Markdown headings, not from the API response

**Decision:** Parse the returned Markdown string client-side using a simple regex (`/^#{1,3}\s+(.+)/gm`) to extract headings and generate anchor links. Do not add a `toc` field to the API response.

**Rationale:** Keeps the API contract minimal. The Markdown already contains the canonical headings; deriving TOC from it avoids duplication and ensures TOC is always in sync.

**Anchor strategy:** Each `<h2>` and `<h3>` in the rendered Markdown gets an `id` attribute generated by slugifying the heading text: lowercase, spaces→hyphens, strip special chars. The TOC links use `href="#<slug>"`. `react-markdown` supports custom renderers via `components` prop.

### D3 — Export: client-side only, no server round-trip

**Decision:** "Copy Markdown" writes `result.markdown` to clipboard. "Download .md" creates a Blob URL. "Download openapi.json" serializes `result.openapi` to a Blob URL. All happen entirely in the browser.

**Rationale:** No new API endpoints needed. Simple, fast, works offline once the result is loaded.

### D4 — localStorage key: `gap:docs-generator:last-result`

**Decision:** After a successful generation, write `{ markdown, openapi, timestamp }` to `localStorage` under this key. On page mount, read it and pre-populate the result panel if present (show a "loaded from cache — generated at <time>" banner). User can dismiss or re-generate.

**Rationale:** Hackathon context — judges may accidentally refresh. localStorage is the simplest persistence that survives a reload without requiring DB or auth.

### D5 — API route: `POST /api/docs`

**Decision:** Create `app/api/docs/route.ts` if not present. Body: `{ backendCode: string }`. Response: `{ analysisId: string | null, markdown: string, openapi: object }`. The route calls `parseBackendRoutes` → `classifyFeatures` → `buildDocumentedRoutes` → `generateApiDocs` in sequence. It attempts to persist to DB via Prisma; if DB is down, it returns the result without an `analysisId` (null).

**Rationale:** Graceful degradation is important for a hackathon demo — a DB blip must not kill the UI.

### D6 — Loading step messages: client-driven, time-based

**Decision:** The page shows 4 rotating step messages while awaiting the API response. Messages advance on a timer (every 5 s) regardless of actual pipeline progress — the API is a single awaited call.

**Rationale:** The backend pipeline does not stream progress events. A fake-but-informative progression makes the UX feel faster and more alive. Each step is clearly a UI affordance, not a real status.

## Risks / Trade-offs

- **Large Markdown output:** Gemini may return very large Markdown (10k+ chars). `react-markdown` handles this fine, but the TOC regex must be efficient. → No mitigation needed beyond basic profiling.
- **`rehype-highlight` bundle size:** Adds ~100 KB gzipped (highlight.js). → Acceptable for a hackathon; can code-split if needed.
- **localStorage 5 MB limit:** Markdown + OpenAPI JSON could theoretically exceed this for huge codebases. → Catch `QuotaExceededError` silently; skip persistence if it fails.
- **OpenAPI field from API may be null:** The `generateApiDocs` function may not always return a valid OpenAPI object. → Disable the "download openapi.json" button if `openapi` is null/undefined.
- **No streaming — long wait:** A large codebase may take 30–60 s for the full pipeline. → The spinner + step messages manage perception; set `timeout` in the API route to 90 s.

2. Import `highlight.js/styles/github-dark.css` in `app/globals.css`.
3. Create `app/api/docs/route.ts` (or verify it exists and matches the contract above).
4. Create `components/ApiDocPanel.tsx`.
5. Create `app/docs-generator/page.tsx`.
6. Test locally with sample Express code (Sample A from CLAUDE.md).
7. Verify localStorage persistence across reload.

Rollback: all changes are additive; deleting the new files is sufficient.
